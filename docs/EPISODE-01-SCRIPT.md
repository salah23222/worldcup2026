📷 **كاميرا — اشرح بثقة**

> «**ثلاث قرارات أكثر مهمّة في المشروع**:»

### القرار 1: PHP خام بدون framework

> «Laravel و Symfony أدوات رائعة. لكن لكل مشروع. لمشروع شخصي
> صغير، الـ overhead الزائد يكسر السرعة. PHP خام يعني:
> - **زيرو تبعيّات** — composer install ما عمري احتاجه
> - **زيرو تحديثات breaking** — اللي بنيته الأسبوع الماضي يعمل
> - **استضافة أي مكان** — أي shared hosting بـ $2/شهر يكفي
> - **سيطرة كاملة** — لو شي ما يعمل، أعرف ليش، حالاً»

### القرار 2: ملف JSON بدل DB في البداية

🖥️ **افتح `cache/worldcup.json` وأعرض حجم الملف**

> «openfootball يعطيني ملف JSON يحوي كل بطولات العالم بالكامل.
> 104 مباراة. حجمه 80 كيلوبايت. PHP يقرأه في ميلي ثانية واحدة.
> لماذا أنشئ DB لشيء ثابت يتغيّر مرّة كل 4 سنين؟»
>
> «المرحلة الثانية: لمّا أضفت **الحسابات والتوقّعات**، احتجت
> MySQL. لكن للبيانات الرياضيّة، بقي JSON.»

### القرار 3: لا framework لـ frontend

🖥️ **افتح أدوات المطوّر → Lighthouse**

> «اضغطوا run audit. شوفوا. Performance 95+. Accessibility 95+.
> أربع كيلوبايت JavaScript فقط. لو استخدمت React، الـ bundle
> راح يكون 200 كيلوبايت كحد أدنى.»

> «الموقع **يفتح في ثانيتين** على 3G. هذي الفرق.»

## ⏱️ [4:00 — 6:30] الهندسة العامّة — المعمارية في صورة

🖥️ **افتح Excalidraw أو ارسم على iPad**

📐 **ارسم هذا الـ flow ببطء**:

```
[المتصفّح]
    ↓
[Hostinger Apache]
    ↓
[bootstrap.php] ← يُحمّل كل شيء
    ↓
[الصفحة الـ PHP] ← مثل matches.php
    ↓
[Service Classes] — DataService, Predictions, AiContent, ...
    ↓
[المصادر]:
  - openfootball JSON (cache 5 دقائق)
  - MySQL (للحسابات والتوقّعات)
  - Claude API (للتقارير، cached)
  - API-Football (للنتائج اللحظيّة)
```

> «كل صفحة في الموقع تتبع نفس هذا المسار. السطر الأوّل في
> كل صفحة: require bootstrap. والذي بدوره يحمّل كل الخدمات.»

🖥️ **افتح أيّ صفحة، مثلاً `matches.php`**

> «شفتوا؟ السطر الأوّل: require. السطر الأخير: tpl footer.
> بينهما الكود الذي يخصّ هذه الصفحة فقط. هذا اتساق وسهولة قراءة.»
